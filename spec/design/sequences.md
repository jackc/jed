# Sequences ‚Äî design

> `CREATE SEQUENCE name [options]` / `DROP SEQUENCE` and the value functions `nextval('s')`,
> `currval('s')` (later `setval`/`lastval`): named, persisted, monotonic **i64 generators** ‚Äî the
> PostgreSQL sequence object. A sequence is a database-level catalog object (like a composite type),
> created and dropped at runtime, persisted in the catalog, and advanced by `nextval`. This doc is
> the contract all three cores implement in lockstep (CLAUDE.md ¬ß2); the grammar is in
> [../grammar/grammar.ebnf](../grammar/grammar.ebnf) + [grammar.md](grammar.md), the value functions
> in [../functions/catalog.toml](../functions/catalog.toml) + [functions.md](functions.md), the byte
> layout in [../fileformat/format.md](../fileformat/format.md) (`format_version` 12), the cost
> contract in [cost.md](cost.md), and the errors in [../errors/registry.toml](../errors/registry.toml).
> PostgreSQL semantics were pinned against the live `postgres:18` oracle (CLAUDE.md ¬ß1).

A sequence is the third kind of **database-level catalog object** (after tables and composite types)
and the first object whose entire reason for existing is to carry **mutable state** ‚Äî a counter that
`nextval` advances. That state lives in the snapshot catalog, so a sequence's current value is part
of the committed database image and moves through the ¬ß3 staging-buffer / commit machinery exactly
like a table's rows. The one decision that defines the feature is what happens to that advance on
`ROLLBACK` ‚Äî see ¬ß5.

## 1. Surface

```sql
CREATE SEQUENCE s
CREATE SEQUENCE [IF NOT EXISTS] s
    [AS { smallint | integer | bigint }]        -- the value type (S5 ‚Äî ¬ß14); default bigint
    [INCREMENT [BY] n] [MINVALUE m | NO MINVALUE] [MAXVALUE x | NO MAXVALUE]
    [START [WITH] s0] [CACHE c] [[NO] CYCLE]
ALTER SEQUENCE [IF EXISTS] s RESTART [WITH n]   -- reset the counter (S2 ‚Äî ¬ß4)
ALTER SEQUENCE [IF EXISTS] s                     -- change the definition (S6 ‚Äî ¬ß15): the order-free
    [INCREMENT [BY] n] [MINVALUE m | NO MINVALUE] -- CREATE option set (minus AS) + RESTART, ‚â•1 action
    [MAXVALUE x | NO MAXVALUE] [START [WITH] s0]
    [RESTART [[WITH] r]] [CACHE c] [[NO] CYCLE]
ALTER SEQUENCE [IF EXISTS] s RENAME TO s2        -- rename within the relation namespace (S6 ‚Äî ¬ß15)
DROP SEQUENCE [IF EXISTS] s [, ...] [RESTRICT]

CREATE TABLE t (id serial, ...)                 -- the serial pseudo-types (S3 ‚Äî ¬ß12): an
CREATE TABLE t (id bigserial, ...)              -- auto-created OWNED sequence + DEFAULT
CREATE TABLE t (id smallserial, ...)            -- nextval(...) + NOT NULL; DROP TABLE auto-drops it
CREATE TABLE t (id int GENERATED ALWAYS AS IDENTITY, ...)      -- identity columns (S4 ‚Äî ¬ß13):
CREATE TABLE t (id int GENERATED BY DEFAULT AS IDENTITY, ...)  -- serial's machinery + ALWAYS/BY DEFAULT
INSERT INTO t (id, v) OVERRIDING SYSTEM VALUE VALUES (1, 'x')  -- gating + the OVERRIDING clause (¬ß13)

SELECT nextval('s')            -- advance and return the new value (a WRITE statement ‚Äî ¬ß4)
SELECT currval('s')            -- the last value nextval/setval produced IN THIS SESSION (¬ß6)
SELECT setval('s', n)          -- set the counter; next nextval = n + INCREMENT (a WRITE ‚Äî ¬ß4)
SELECT setval('s', n, false)   -- set the counter; next nextval = n (is_called = false)
SELECT lastval()               -- the value the most recent nextval returned this session (¬ß6)
```

- **i64-valued, with an `AS` data type (S5 ‚Äî ¬ß14).** A sequence's counter is always an `i64`
  internally (`nextval`/`currval` return `i64`); the **`AS smallint | integer | bigint`** clause (PG
  10+, an order-free option) selects the value **type**, which sets the **default MINVALUE/MAXVALUE**
  and **bounds** any explicit ones. `smallint` ‚Üí `[-32768, 32767]`, `integer` ‚Üí `[-2147483648,
  2147483647]`, `bigint` (the default, with no `AS`) ‚Üí `[-2^63, 2^63-1]` (the full `i64` range).
  A non-integer `AS` type (`AS text`, `AS numeric`, ‚Ä¶) is `22023` ("sequence type must be smallint,
  integer, or bigint"). The type is **not persisted** ‚Äî it is fully reducible to the resulting
  MIN/MAX bounds (¬ß14), so a typed sequence and a plain one with the same explicit bounds are
  byte-identical; no `format_version` change.
- **Options.** `INCREMENT BY` (non-zero step; `22023` for zero), `MINVALUE`/`MAXVALUE` (the inclusive
  bounds; `NO MINVALUE`/`NO MAXVALUE` select the **type** defaults), `START WITH` (the first value),
  `CACHE`, `[NO] CYCLE` (wrap vs. error at a bound), and `AS <type>` (above). Defaults match
  PostgreSQL relative to the chosen type `T` (`[T_min, T_max]`): a positive `INCREMENT` (default `1`)
  gives `MINVALUE 1`, `MAXVALUE T_max`, `START` = `MINVALUE`; a negative `INCREMENT` gives `MINVALUE
  T_min`, `MAXVALUE -1`, `START` = `MAXVALUE`. An explicit `MAXVALUE > T_max` is `22023` ("MAXVALUE
  (x) is out of range for sequence data type *T*") and an explicit `MINVALUE < T_min` is `22023`
  ("MINVALUE (x) is out of range ‚Ä¶") ‚Äî checked before the consistency checks. A
  `START`/`MINVALUE`/`MAXVALUE` combination that is inconsistent (`START < MINVALUE`, `START >
  MAXVALUE`, or `MINVALUE > MAXVALUE`) is `22023` at `CREATE`.
- **`CACHE`** is **parsed and stored but behaviorally `1`** ‚Äî see ¬ß7. A `CACHE < 1` is `22023`.
- **`nextval('s')`/`currval('s')`/`setval('s', n[, is_called])`** take the sequence **name as a text
  argument** (the PG `nextval('s'::regclass)` form, with the `regclass` cast implicit). They are the
  first built-in functions to resolve a string argument to a **catalog object** and (`nextval`/
  `setval`) the first to **mutate** the database during expression evaluation. **`lastval()`** takes
  no argument (the first 0-arg sequence function). `setval`/`lastval` land in **S2** (¬ß11) ‚Äî their
  precise semantics are ¬ß4 (`setval`) and ¬ß6 (`lastval`).
- **`DROP SEQUENCE`** is `RESTRICT` by default and RESTRICT-only this slice (`CASCADE` is `0A000`).
  A missing sequence is `42P01` (sequences share the relation namespace ‚Äî like `DROP TABLE`), unless
  `IF EXISTS`. **No dependency tracking this slice:** a plain `column DEFAULT nextval('s')` does *not*
  create a dependency in PostgreSQL either (only `serial`/`OWNED BY`/identity do, both deferred), so
  `DROP SEQUENCE` of a sequence named in some column default succeeds and a later `INSERT` raises
  `42P01` at evaluation ‚Äî exactly PG.
- **`ALTER SEQUENCE [IF EXISTS] s RESTART [WITH n]`** landed in **S2** (¬ß4). **S6 (¬ß15)** broadens
  `ALTER` to the **definition-changing option set** ‚Äî the *same* order-free options as `CREATE`
  (`INCREMENT`/`MINVALUE`/`MAXVALUE`/`START`/`CACHE`/`[NO] CYCLE`), plus `RESTART` as an interleavable
  pseudo-option ‚Äî and to **`RENAME TO`**. An `ALTER` with **no action** is `42601` (PG: bare `ALTER
  SEQUENCE s` is a syntax error). Still `0A000`: **`AS type`** (the type is not persisted ‚Äî ¬ß14.4 ‚Äî so
  re-deriving the default bound for a new type is foreclosed), **`OWNED BY`**, **`OWNER TO`**, and
  **`SET { SCHEMA | LOGGED | UNLOGGED }`**.

## 2. Sequences as a catalog object ‚Äî `Snapshot.sequences`

A sequence is resident in the snapshot catalog beside `tables` and `types`:

- A new `sequences` map on `Snapshot` (Rust `HashMap<String, SequenceDef>`, Go `map[string]*SequenceDef`,
  TS `Map<string, SequenceDef>`), keyed by **lowercased name** ‚Äî the same keying tables/types use.
- `SequenceDef = { name, increment, min_value, max_value, start, cache, cycle, last_value, is_called }`
  ‚Äî all `i64` except `name`, `cycle`, and `is_called`. `last_value` + `is_called` are the **mutable
  counter state** (PG's sequence-tuple fields); the rest are immutable definition (PG's `pg_sequence`
  fields). On `CREATE`, `last_value = start` and `is_called = false`.
- Because the whole `Snapshot` is the unit a commit publishes and a rollback discards (CLAUDE.md ¬ß3,
  [transactions.md ¬ß4.5](transactions.md)), the counter is **transactional by construction** ‚Äî no
  separate store, no seam, no special casing (¬ß5).

The name occupies the **relation namespace** it shares with tables and indexes: `CREATE SEQUENCE s`
when a table, index, or sequence `s` exists is `42P07 duplicate_table` (PG), and a `CREATE TABLE s`
after `CREATE SEQUENCE s` is likewise `42P07`. (This slice enforces the sequence‚Üîsequence and
sequence‚Üîtable collisions; a sequence‚Üîindex collision rides the same name set.)

## 3. Catalog & on-disk format (`format_version` 12, owner added in `14`, identity bits in `15`)

Sequences extend the v9 **kind-tagged** catalog (today `0` = table, `1` = composite-type) with a
third kind. The on-disk shape (full byte layout in [../fileformat/format.md](../fileformat/format.md)):

- **`entry_kind = 2`** is a sequence entry. Emission order across the catalog is **composite-type
  entries (kind 1) first, then sequence entries (kind 2), then table entries (kind 0)**, each group in
  ascending lowercased-name order; `item_count` per catalog page counts all entries, packed greedily
  exactly as before. Tables stay last because a column may reference a composite type by name at load;
  a sequence is referenced by **nothing** at load (a `DEFAULT nextval('s')` is stored expr-text,
  resolved at evaluation), so sequence entries need no two-pass and may sit anywhere ahead of the
  tables ‚Äî they are grouped with the other non-table objects for a clean "schema objects, then tables"
  layout.
- **Sequence entry** (after the `entry_kind = 2` byte): `name_len u16` + name, then six fixed
  `i64` fields **big-endian two's-complement, no sign-flip** (a value-codec context, not a key ‚Äî
  like the interval body) in this order ‚Äî `increment`, `min_value`, `max_value`, `start`, `cache`,
  `last_value` ‚Äî then `flags u8` (bit 0 = `cycle`, bit 1 = `is_called`, **bit 2 = `has_owner`** ‚Äî
  new in v14; bits 3‚Äì7 reserved, written 0). The six i64 fields are fixed-width, no presence tags:
  every one is always present and non-NULL. **When `has_owner` is set** (an `owned` sequence ‚Äî a
  `serial` column's sequence, ¬ß12), the flags byte is followed by the owner reference:
  `owner_table_len u16` + owner table name, then `owner_column u16` (the owning column's **0-based
  ordinal**). A non-owned sequence (a plain `CREATE SEQUENCE`) writes nothing after the flags byte ‚Äî
  the v12 shape, unchanged.
- **`format_version` 15** (S4, ¬ß13) adds **no sequence-entry change** ‚Äî an IDENTITY column is a
  `serial`-shaped owned sequence; the only on-disk change is two bits in the *column* entry's `flags`
  byte (bit 4 `is_identity`, bit 5 `identity_always` ‚Äî format.md *Catalog*). The `identity_table.jed`
  golden pins it (`rust == go == ts == ruby`).
- **`format_version` 12** was the original sequence bump; **`format_version` 14** adds the optional
  owner reference (the only on-disk change ‚Äî bit 2 + the trailing owner bytes). Each is a clean break
  (the v14 reader rejects v13 and earlier, as every prior bump did). All existing `.jed` goldens
  regenerate at a bump (only the version byte changes for a file with no owned sequences; an owned
  sequence gains the trailing owner bytes). The reference encoder
  ([../fileformat/verify.rb](../fileformat/verify.rb)) carries the `sequences:` fixture key, the
  `sequence_table.jed` golden, and the v14 `serial_table.jed` golden, byte-identical
  `rust == go == ts == ruby`.

A sequence entry adds no value-codec, key-encoding, or B-tree change ‚Äî a sequence owns no rows and no
tree, only its catalog tuple (like a `FOREIGN KEY`, which also owns no B-tree).

## 4. The value functions ‚Äî name resolution, the write implication, read-only

`nextval`/`currval` are ordinary `[[operator]]` catalog rows (`kind = "function"`, `arg_families =
["text"]`, `result = "i64"`, `null = "propagates"`, `volatility = "volatile"`), resolved by the
normal overload path. Three things make them the first of their kind:

- **Name ‚Üí catalog object at evaluation time.** The argument is a `text` value naming a sequence. The
  evaluator (which already holds the executor/snapshot ‚Äî the same access `now()` uses for the clock
  seam) looks the name up in `Snapshot.sequences` (case-insensitively, the catalog keying). A missing
  sequence is `42P01 undefined_table` (PG renders "relation \"s\" does not exist"). A **NULL** argument
  propagates NULL (the `null = "propagates"` discipline) ‚Äî matching PG, where `nextval(NULL)` is NULL.
- **`nextval`/`setval` mutate, so the statement is a WRITE.** A `SELECT nextval('s')` advances the
  sequence and therefore must run on the **write path**: take the single-writer gate, stage the
  advanced `SequenceDef` into the working snapshot, and commit it (autocommit) or carry it in the open
  transaction. The executor's write-detection (`stmt_is_write`) is extended: a statement is a write if
  its resolved expression tree contains a **sequence-mutating** function (`nextval`; `setval` in S2).
  `currval` is pure-read (session state only) and never forces the write path.
- **Read-only transactions.** `nextval`/`setval` inside a `READ ONLY` transaction (or a read handle)
  is `25006 read_only_sql_transaction` (PG: "cannot execute nextval() in a read-only transaction").
  `currval` is allowed in a read-only transaction.

**`nextval` semantics** (PG-exact, [pg `sequence.c`]): on a sequence with state `(last_value,
is_called)`:

- if `!is_called`: the result is `last_value` (the `START` on a fresh sequence); set `is_called =
  true`. The counter is *not* incremented ‚Äî the first `nextval` returns `START`.
- else: compute `next = last_value + increment`. If `increment > 0` and `next > max_value`: if `cycle`,
  `next = min_value`, else `2200H sequence_generator_limit_exceeded` ("nextval: reached maximum value
  of sequence \"s\""). Symmetrically for `increment < 0` / `next < min_value`. The result is `next`;
  set `last_value = next`. The add is **overflow-safe** (a wrap past the i64 boundary is treated as
  crossing the bound, never a native overflow).

After a successful `nextval`, the value is recorded in the **session** state (¬ß6) for `currval` and
`lastval`.

**`setval` semantics** (PG-exact, [pg `sequence.c` `do_setval`]): `setval('s', n)` and `setval('s',
n, is_called)` set the counter directly. The value `n` must lie in `[min_value, max_value]` or it is
`22003 numeric_value_out_of_range` ("setval: value n is out of bounds for sequence \"s\"
(min..max)"). On success: `last_value = n` and `is_called` = the third argument (default `true`); the
result is `n`. The effect on the next `nextval` follows from `is_called` ‚Äî `setval('s', n)` (called)
makes the next `nextval` return `n + increment`; `setval('s', n, false)` makes it return `n`.
`setval` is a **write** (the write path + `25006` in a read-only txn, exactly like `nextval`) and is
**transactional** (rolls back ‚Äî ¬ß5). Two deliberate session-state asymmetries with `nextval`, both
matching PG (verified against the oracle): (a) `setval` updates `currval` **only when `is_called` is
true** (with `is_called = false`, `currval` keeps its prior value / stays `55000`); (b) `setval`
**never** updates `lastval` ‚Äî `lastval` tracks `nextval` alone (¬ß6).

**`ALTER SEQUENCE [IF EXISTS] s RESTART [WITH n]`** resets the counter: `last_value = n` (`RESTART
WITH n`) or `last_value = start` (bare `RESTART`, the original `START` ‚Äî `RESTART` does **not** change
the stored `start`), with `is_called = false`, so the next `nextval` returns that value. A value
outside `[min_value, max_value]` is `22023 invalid_parameter_value` (PG: "RESTART value (n) cannot be
greater than MAXVALUE (max)" / "‚Ä¶ less than MINVALUE (min)" ‚Äî note `22023`, **not** `setval`'s
`22003`). A missing sequence is `42P01` unless `IF EXISTS` (then a no-op). `ALTER` is a catalog
mutation (the write path, transactional) and touches **no** session state ‚Äî `currval`/`lastval` are
unchanged by a `RESTART`.

**`ALTER SEQUENCE [IF EXISTS] s <options>`** (S6 ‚Äî ¬ß15) re-runs PG's `init_params` against the
*existing* definition (`isInit = false`): only the **explicitly written** options change; everything
else ‚Äî including `last_value`/`is_called` ‚Äî is **preserved**. So `ALTER SEQUENCE s INCREMENT BY 10` on
a sequence whose `last_value` is `2` leaves the counter at `2` and the *next* `nextval` returns `12`.
`RESTART` may be interleaved with the options (`ALTER ‚Ä¶ INCREMENT 100 RESTART WITH 50 MAXVALUE 1000`)
and is applied last. The same `22023` validations as `CREATE` apply (`INCREMENT ‚âÝ 0`, `CACHE ‚â• 1`,
`MINVALUE < MAXVALUE` ‚Äî strict, PG rejects `MINVALUE == MAXVALUE`), plus the **two cross-checks PG runs
against the post-edit bounds**, in this order: (1) `START ‚àà [MINVALUE, MAXVALUE]` ("START value (s)
cannot be less than MINVALUE (m)" / "greater than MAXVALUE (m)"), then (2) the **current `last_value`**
(after any `RESTART`) `‚àà [MINVALUE, MAXVALUE]` ("RESTART value (lv) cannot be ‚Ä¶", PG's wording even
when no `RESTART` was given). `NO MINVALUE` / `NO MAXVALUE` reset to the default for the current
`INCREMENT` sign (ascending ‚Üí `MINVALUE 1`; descending ‚Üí `MAXVALUE -1`); the *open* direction resets to
the **bigint** bound because the value type is not persisted (¬ß14.4 / ¬ß15) ‚Äî a documented divergence
for a typed sequence only. Changing the `INCREMENT` sign **does not** recompute the bounds (PG keeps
them), so a previously-ascending sequence altered to a negative `INCREMENT` keeps its `MINVALUE 1`.

**`ALTER SEQUENCE [IF EXISTS] s RENAME TO s2`** (S6 ‚Äî ¬ß15) renames the sequence within the relation
namespace: `s2` already naming a table/index/sequence (including `s` itself) is `42P07
duplicate_table` ("relation already exists"). The counter and all options are unchanged. For an
**owned** sequence (a `serial`/identity column's, ¬ß12) the owning column's `DEFAULT nextval('s')`
expression text is **rewritten** to `nextval('s2')` in the same statement, so a subsequent `INSERT`
still advances the renamed sequence ‚Äî matching PG's observable behavior (PG references the sequence by
OID, jed by the now-rewritten name; the `owned_by` link itself, a `(table, column-ordinal)` pair, is
unchanged so `DROP TABLE` auto-drop and the `2BP01` on a direct drop still hold). A missing sequence is
`42P01` unless `IF EXISTS`.

## 5. The defining decision ‚Äî transactional sequences (a documented PG divergence)

> **jed sequences are transactional: `nextval` rolls back with its transaction.** This is a
> deliberate divergence from PostgreSQL, where sequence advances are **non-transactional** (never
> rolled back; gaps are allowed) ‚Äî and it is already mandated by
> [determinism.md ¬ß5](determinism.md): "Sequences / `SERIAL` / identity columns ‚Ä¶ are **fully
> deterministic** (a monotonic counter, reconstructed on load) and stay inside the contract. **Do not
> exempt them.**"

The reasoning, recorded here and in the override ledger:

- **PG's non-transactionality is a concurrency optimization jed does not need.** PG lets `nextval`
  escape transaction rollback so concurrent sessions never block on a sequence ‚Äî the cost is gaps. jed
  is **single-writer** (CLAUDE.md ¬ß3): there is no concurrent contention on the counter to optimize
  away, so the only thing PG's gaps would buy us is nondeterminism.
- **It would break the cross-core determinism contract.** A non-transactional counter advancing across
  rollback is observable mutable state outside the snapshot; to keep `rust == go == ts` byte-identical
  it would need a host **seam** and a **determinism-ledger exemption**
  ([determinism_exceptions.toml](../conformance/determinism_exceptions.toml)) ‚Äî exactly what
  determinism.md ¬ß5 forbids for counters. Transactional sequences need **neither**: the counter is an
  ordinary snapshot field, deterministic by construction, reconstructed from disk on open.
- **What is observably different from PG:** after `BEGIN; SELECT nextval('s'); ROLLBACK;`, jed's `s` is
  unchanged; PG's `s` keeps the advance. After a failed statement inside a transaction, same. Within a
  *committed* transaction the behavior is identical to PG (the advance persists, `currval` sees it).
  Two successful `nextval`s in autocommit also match PG (each commits). The divergence surfaces
  **only** on rollback/abort.

This makes the entire feature deterministic with no new seam: a sequence's value is a pure function of
`(CREATE options, the committed sequence of nextval/setval calls)`, identical on every core.

## 6. `currval` / `lastval` ‚Äî per-session state, not snapshot state

`currval('s')` returns the value most recently produced by `nextval('s')` **in the current session**
(handle) ‚Äî PostgreSQL's strictly session-local semantics, **independent of other sessions and of the
committed sequence value**. This is **per-handle transient state**, NOT part of the snapshot and NOT
persisted:

- Each `Database` handle carries a small `session_seq: map<lowercased-name ‚Üí i64>` of the last value
  this handle's `nextval`/`setval` produced for each sequence, plus a single `session_last: i64?` ‚Äî
  the most-recent-overall value `nextval` returned (the `lastval` source).
- `currval('s')` before any `nextval('s')`/`setval('s', ‚Ä¶, true)` in this session is `55000
  object_not_in_prerequisite_state` ("currval of sequence \"s\" is not yet defined in this session") ‚Äî
  even if the sequence exists and another session advanced it. A missing sequence is still `42P01`
  (the name is resolved against the catalog first).
- **`lastval()`** returns `session_last` ‚Äî the value the most recent `nextval` (of **any** sequence)
  returned in this session ‚Äî and is `55000 object_not_in_prerequisite_state` ("lastval is not yet
  defined in this session") before the first `nextval`. It takes **no name argument** (no `42P01`
  path). Two oracle-pinned details: `lastval` reads `nextval`'s history **only** (a `setval` never
  updates `session_last`, so `lastval` is unaffected by it), and `currval`'s per-sequence map **is**
  updated by `setval('s', n)` (called) but **not** by `setval('s', n, false)`.
- Because jed read results depend only on commit order + the session's own call history (never wall
  clock), `currval` is deterministic *given the session's statement sequence* ‚Äî the conformance corpus
  is single-handle and sequential, so it pins `currval` directly; the cross-session independence is a
  per-core / concurrency-suite concern.
- **Does `currval`/`lastval` survive rollback?** jed records the session values (`session_seq`,
  `session_last`) on a *successful* `nextval`/`setval` evaluation, flushed together with the counter
  advance on statement success. Since a rolled-back `nextval`/`setval` did not commit and (jed ¬ß5) did
  not advance the sequence, the session values it set are also discarded on rollback, keeping
  `currval`/`lastval` consistent with the transactional counter. (Moot under PG-comparison since PG's
  session value *would* survive ‚Äî another facet of the ¬ß5 divergence, ledgered.)

## 7. `CACHE` ‚Äî accepted, never value-burning (a documented PG divergence)

PostgreSQL's `CACHE c` pre-allocates `c` values to a session at once, so values are consumed in
per-session blocks and **gaps appear** when a session exits with cache unused (and the cached block is
non-transactional). That is two nondeterminism sources at once (cross-session block interleaving + gap
on disconnect), incompatible with the ¬ß5 determinism stance. jed therefore **parses and stores
`CACHE c`** (so the clause is accepted and round-trips for fidelity and future `ALTER`) but
**behaves as `CACHE 1`**: every `nextval` advances the single shared counter by exactly `increment`,
no per-session reservation, no gaps. Recorded as a divergence. (`CACHE` < 1 is `22023`.)

## 8. Cost

A new `sequence_advance` cost unit ([../cost/schedule.toml](../cost/schedule.toml)) is charged once
per `nextval`/`setval` evaluation, **in addition** to the one `operator_eval` every function call
already rides ‚Äî the advance touches and rewrites a catalog tuple, more than a pure value‚Üívalue map.
`currval` is a pure session-state read and charges only its `operator_eval`. The unit keeps a runaway
`nextval` (e.g. a `generate_series`-driven sweep) bounded by `max_cost` (CLAUDE.md ¬ß13, the deterministic
ceiling) and keeps cost cross-core identical (CLAUDE.md ¬ß8). Catalog-tuple I/O is schema-bounded, so a
flat per-call weight is a sound bound.

## 10. Ratified decisions and deliberate PostgreSQL divergences

Default is "match PostgreSQL" (CLAUDE.md ¬ß1); the divergences below each have an overriding reason and
are recorded in [../conformance/oracle_overrides.toml](../conformance/oracle_overrides.toml).

1. **Transactional sequences** ‚Äî `nextval` rolls back with the transaction (¬ß5). Overriding reason:
   determinism (CLAUDE.md ¬ß8/¬ß10, determinism.md ¬ß5) + the single-writer model removes PG's
   concurrency rationale. The headline divergence.
2. **`CACHE` is no value-burning** ‚Äî accepted and stored, behaves as `CACHE 1` (¬ß7). Same reason.
3. **~~`bigint`-only~~ ‚Äî CLOSED in S5 (¬ß14).** `AS smallint | integer | bigint` is now supported; a
   sequence's value type sets its default/validated bounds (the counter stays `i64` internally). The
   type is not persisted (it is reducible to the bounds ‚Äî ¬ß14). `ALTER ‚Ä¶ AS type` stays `0A000`.
4. **No implicit dependency from a plain `DEFAULT nextval('s')`** ‚Äî matches PG (only `serial`/identity
   create one); `DROP SEQUENCE` needs no dependency tracking this slice (¬ß1).
5. **`nextval`/`setval` make the statement a write** ‚Äî required by the single-writer staging model
   (¬ß4); a `SELECT nextval('s')` commits a new snapshot. Observably matches PG (the value persists).
6. **Session-local `currval`/`lastval` with `55000`** ‚Äî adopted as-is (¬ß6).
7. **`setval`/`ALTER RESTART` are transactional** ‚Äî they roll back like `nextval` (¬ß5, the same
   reason). Their session-state asymmetries match PG exactly (`setval(‚Ä¶,false)` leaves `currval`
   alone; `setval` never touches `lastval`; `RESTART` touches no session state ‚Äî ¬ß4/¬ß6).
8. **`setval` out-of-bounds is `22003`, `RESTART` out-of-bounds is `22023`** ‚Äî the two distinct PG
   error paths (`do_setval` vs. `init_params`) are preserved as-is (¬ß4).
9. **~~`serial` sequences are `bigint`-flavored~~ ‚Äî CLOSED in S5 (¬ß14).** With `AS type` implemented,
   a `serial`/`bigserial`/`smallserial` column's owned sequence now follows the pseudo-type
   (`serial` ‚Üí `AS integer`, `bigserial` ‚Üí `AS bigint`, `smallserial` ‚Üí `AS smallint`), matching PG
   exactly ‚Äî a too-large `nextval` traps `2200H` at the sequence (PG's behavior), not `22003` at the
   INSERT coercion. The divergence is removed (no oracle override).
10. **The owned (`serial`) sequence is dropped with its table; dropping it alone is `2BP01`** (¬ß12) ‚Äî
    the `OWNED BY` dependency. Matches PG. Unlike a *plain* `DEFAULT nextval('s')` (decision 4, no
    dependency), a `serial`-created sequence carries a persisted owner link.
11. **~~IDENTITY sequences are `bigint`-flavored~~ ‚Äî CLOSED in S5 (¬ß14).** A `GENERATED ‚Ä¶ AS IDENTITY`
    column's owned sequence now follows the **column type** (`i16` ‚Üí `AS smallint`, `i32` ‚Üí `AS
    integer`, `i64` ‚Üí `AS bigint`), matching PG ‚Äî the sequence itself is bounded to the column's range
    (a too-large `nextval` traps `2200H` at the sequence). An explicit `AS` inside the identity `(
    ‚Ä¶ )` options is `42601` ("conflicting or redundant options" ‚Äî the column type already fixes it).
    Identity columns are otherwise behaviorally PG-exact: ALWAYS/BY DEFAULT gating, `OVERRIDING`, the
    `42601` conflicts, the `22023` type gate, the owned-sequence auto-drop / `2BP01` (decision 10).

  determinism divergence recorded. The decisions are ratified spec-first before any core changes.
- **S1** ‚Äî `CREATE SEQUENCE` (full option grammar) / `DROP SEQUENCE` + the `sequences` catalog map +
  `format_version` 12 + the `sequence_table.jed` golden (`rust == go == ts == ruby`) + `nextval` +
  `currval` + the `sequence_advance` cost unit + the write-path detection + the read-only `25006` +
  the conformance corpus (`ddl/sequence.test`, `expr/sequence_value.test`) + capabilities
  `ddl.sequence` / `func.sequence`. The "it's alive" slice ‚Äî a sequence is created, advanced, read,

- **S2** ‚úÖ ‚Äî `setval(s, n[, is_called])` + `lastval()` + `ALTER SEQUENCE [IF EXISTS] s RESTART

  statement** and then asserts a single `nextval`/`currval`, so the S1 "advance via `statement ok`,
  read via terminal `query`" scaffolding is replaced by direct `setval`-then-assert checks.
- **S3** ‚úÖ ‚Äî the `serial` / `bigserial` / `smallserial` pseudo-types (¬ß12): column sugar that, at
  `CREATE TABLE`, creates an **owned** sequence + a `DEFAULT nextval(...)` + `NOT NULL` and records
  the `OWNED BY` link, so `DROP TABLE` auto-drops the owned sequence and `DROP SEQUENCE` of an owned

- **S4** ‚úÖ ‚Äî `GENERATED { ALWAYS | BY DEFAULT } AS IDENTITY` columns (the SQL-standard identity

  clause. Reuses S3's owned-sequence + `nextval`-default + `NOT NULL` desugaring wholesale, adding only
  two persisted column flag bits (**`format_version` 15** ‚Äî bit 4 `is_identity`, bit 5
  `identity_always`, ¬ß3), the `identity_table.jed` golden (`rust == go == ts == ruby`), the

  Capability `ddl.identity`; corpus `ddl/identity.test`.
- **S5** ‚Äî the `AS { smallint | integer | bigint }` sequence data type (¬ß14): an order-free `CREATE
  SEQUENCE` option that sets the value type ‚Üí default/validated MIN/MAX bounds; `serial` follows the
  pseudo-type and an IDENTITY column follows its column type (both auto-wiring the owned sequence's
  type). **Closes the bigint-flavored divergence** (decisions 3/9/11) ‚Äî a `smallserial` / `smallint

## 12. `serial` / `bigserial` / `smallserial` ‚Äî the auto-sequence pseudo-types (S3)

`serial` is not a real type: it is **`CREATE TABLE` column sugar** (PostgreSQL) for "an auto-numbered
column backed by its own sequence". jed adopts PG's desugaring exactly. A column declared

```sql
CREATE TABLE t (id serial, ...)
```

is, at `CREATE TABLE` execution, expanded to the equivalent of

```sql
CREATE SEQUENCE t_id_seq;                              -- an OWNED sequence (default i64, ¬ß3)
-- column id  =>  i32 NOT NULL DEFAULT nextval('t_id_seq')
-- the sequence is OWNED BY t.id  (DROP TABLE t auto-drops t_id_seq; ¬ß12.3)
```

### 12.1 The three pseudo-types and their aliases

Recognized **only** in a `CREATE TABLE` column-type position (not in `CAST`, `CREATE SEQUENCE ‚Ä¶ AS`,
a `CREATE TYPE` field, or an array element type ‚Äî there `serial` is an undefined type, the existing
`42704`/‚Äútype does not exist‚Äù path, matching PG):

| Pseudo-type | Aliases | Column type |
|---|---|---|
| `serial` | `serial4` | `i32` (`integer`) |
| `bigserial` | `serial8` | `i64` (`bigint`) |
| `smallserial` | `serial2` | `i16` (`smallint`) |

The pseudo-type sets **only** the column's underlying type; everything else is the desugaring below.
A `serial` column carries **no typmod** and **no `[]` array suffix** ‚Äî `serial(p)` / `serial[]` is a
`42601` syntax-style rejection (jed: `42601`; the natural ‚Äúundefined type‚Äù path also suffices ‚Äî PG
rejects `serial[]` as `0A000` "array of serial is not implemented", a wording divergence not worth a
dedicated branch).

### 12.2 The desugaring (at `CREATE TABLE`)

For each `serial` column `c` of table `t`, in column declaration order:

1. **Column type** ‚Üê `i32` / `i64` / `i16` per the table above.
2. **`NOT NULL`** is forced on the column (PG: a `serial` column is `NOT NULL`). An explicit
   `NOT NULL` is harmlessly redundant; an explicit `NULL` is not a jed surface, so no conflict arises.
3. **An explicit `DEFAULT` is rejected** `42601` ‚Äî "multiple default values specified for column
   *c* of table *t*" (PG): a `serial` column's default is `nextval(...)`, and a second default is an
   error.
4. **The owned sequence** is created: a default ascending sequence (`INCREMENT 1`, `START 1`,
   `CACHE 1`, no `CYCLE`) **of the pseudo-type's data type** (`serial` ‚Üí `AS integer`, `bigserial` ‚Üí
   `AS bigint`, `smallserial` ‚Üí `AS smallint` ‚Äî S5, ¬ß14), so its bounds are `[1, T_max]` (e.g.
   `MAXVALUE 32767` for `smallserial`) ‚Äî the same `SequenceDef` `CREATE SEQUENCE t_id_seq AS <T>`
   would build, **plus** an `owned_by = (t, ordinal-of-c)` link (¬ß3, persisted v14). The sequence
   itself is therefore bounded to the column's range (a too-large `nextval` traps `2200H`, matching
   PG). *(Before S5 the sequence was `bigint`-flavored and the column type alone bounded values.)*
5. **The sequence name** is `lower(t)_lower(c)_seq`. If that name already names a relation
   (table / index / sequence ‚Äî the shared relation namespace, ¬ß2), including a sequence created by an
   *earlier* `serial` column of the **same** statement, the smallest integer suffix `1`, `2`, ‚Ä¶ is
   appended until free (`t_id_seq` taken ‚Üí `t_id_seq1`), matching PG's `ChooseRelationName`.
6. **The column `DEFAULT`** becomes the expression default `nextval ( '<seqname>' )` ‚Äî stored as
   expression text (the `format_version` 8 expression-default mechanism ‚Äî [constraints.md ¬ß2](constraints.md)),
   evaluated per row at INSERT through the existing `nextval` write path (which advances the sequence
   and is transactional, ¬ß4/¬ß5). Supplying an explicit value for the column at INSERT overrides the
   default and does **not** advance the sequence (PG ‚Äî the value and the counter can later collide,
   exactly as in PG).

A `serial` column may be a `PRIMARY KEY` (the common `id serial PRIMARY KEY` form): the `i32`/`i64`/`i16`
type is key-encodable, so the PK gate is satisfied; the column is then PK + `NOT NULL` + the
`nextval` default.

### 12.3 `OWNED BY` ‚Äî the dependency (`DROP TABLE` auto-drop, `DROP SEQUENCE` 2BP01)

The persisted `owned_by` link (¬ß3) is the only thing distinguishing a `serial`-created sequence from a
manually-created one referenced by a plain `DEFAULT nextval('s')` (which creates **no** dependency,
decision 4). It drives two behaviors, both matching PG:

- **`DROP TABLE t`** auto-drops every sequence `owned_by` `t`, silently, as part of the drop (after
  the `FOREIGN KEY`-dependent check, which an owned sequence never trips). The owned sequences are
  removed from the catalog before the table; the drop stays zero-cost. After reopening a persisted
  database the link is still present (v14), so the auto-drop survives a close/open cycle.
- **`DROP SEQUENCE s`** where `s` is owned is `2BP01 dependent_objects_still_exist` ‚Äî
  "cannot drop sequence *s* because other objects depend on it: default value for column *c* of table
  *t* depends on sequence *s*" (RESTRICT; `CASCADE` is `0A000` this slice). The owner table is always
  present when this fires (the table's own `DROP TABLE` would have auto-dropped the sequence first),
  so the column name for the detail is always resolvable.

`DROP TYPE` / `DROP INDEX` are unaffected (a sequence is owned only by a table column). There is no
standalone `OWNED BY` / `ALTER SEQUENCE ‚Ä¶ OWNED BY` surface this slice ‚Äî ownership is established
**only** by `serial` (the `OWNED BY` clause stays `0A000`, ¬ß9's `ALTER` row).

### 12.4 Determinism

`serial` adds no new nondeterminism: the owned sequence is an ordinary snapshot field (transactional,
¬ß5), the auto-generated name is a pure function of `(table, column)` + the catalog (the suffix scan is
deterministic), and the auto-drop is a deterministic catalog edit. The whole feature is byte-identical
cross-core, pinned by the `serial_table.jed` golden (¬ß3) and the `ddl/serial.test` corpus.

## 13. `GENERATED { ALWAYS | BY DEFAULT } AS IDENTITY` ‚Äî identity columns (S4)

Identity columns are the **SQL-standard** spelling of an auto-numbered column. PostgreSQL implements
them (PG 10+) as `serial`'s standardized successor, and jed adopts that surface. The key insight that
keeps this slice small: **an identity column desugars exactly like a `serial` column (¬ß12)** ‚Äî an
**owned** sequence (¬ß12.3) + a `DEFAULT nextval(...)` expression default + `NOT NULL` ‚Äî *plus* one
extra fact persisted on the column, **`GENERATED ALWAYS` vs `GENERATED BY DEFAULT`**, which gates
whether an explicit INSERT/UPDATE value is accepted. So ¬ß13 reuses ¬ß12's machinery wholesale and adds
only: the column-constraint grammar, the optional `( sequence_options )`, two persisted flag bits
(format_version 15, ¬ß3), the `OVERRIDING` clause, and the INSERT/UPDATE value gating.

### 13.1 Surface

```sql
CREATE TABLE t (id integer GENERATED ALWAYS AS IDENTITY, ...)
CREATE TABLE t (id integer GENERATED BY DEFAULT AS IDENTITY, ...)
CREATE TABLE t (id integer GENERATED ALWAYS AS IDENTITY (START WITH 100 INCREMENT BY 5), ...)

INSERT INTO t (id, v) OVERRIDING SYSTEM VALUE VALUES (42, 'x')   -- force an explicit value into ALWAYS
INSERT INTO t (id, v) OVERRIDING USER VALUE   VALUES (42, 'x')   -- discard the value, use the sequence
```

- **The column type must be `smallint` / `integer` / `bigint`** (`i16`/`i32`/`i64`) ‚Äî any other
  declared type is `22023 invalid_parameter_value` ("identity column type must be smallint, integer,
  or bigint"), matching PG (PG raises this from `init_params`). Unlike `serial` (a pseudo-*type*),
  identity is a *constraint* on a written-out real type, so jed validates the resolved column type.
- **The owned sequence** is created exactly as ¬ß12.4's: a default ascending sequence named
  `<table>_<col>_seq` with the same numeric-suffix collision resolution (¬ß12.2 step 5), `OWNED BY` the
  column (persisted v14). Its **data type is the column type** (`i16` ‚Üí `AS smallint`, `i32` ‚Üí `AS
  integer`, `i64` ‚Üí `AS bigint` ‚Äî S5, ¬ß14), so the sequence is bounded to the column's range (a
  `smallint` identity sequence maxes at `32767`, trapping `2200H` past it ‚Äî matching PG). The optional
  **`( sequence_options )`** overrides the sequence definition using the *same* option set and
  validation as `CREATE SEQUENCE` (¬ß1 ‚Äî `INCREMENT`, `MINVALUE`/`MAXVALUE`, `START`, `CACHE`, `[NO]
  CYCLE`; a bad option is the same `22023`, and an explicit bound outside the column type's range is
  `22023` "out of range for sequence data type *T*"). An **`AS`** clause inside these options is
  `42601` ("conflicting or redundant options" ‚Äî the column type already fixes the sequence type).
- **`NOT NULL`** is forced (PG: an identity column is `NOT NULL`).
- **The column `DEFAULT`** becomes the expression default `nextval('<seqname>')` (the v8 mechanism,
  evaluated per row at INSERT through the ¬ß4 write path), identical to a `serial` column's.
- **An identity column may be a `PRIMARY KEY`** (the common `id integer GENERATED ALWAYS AS IDENTITY
  PRIMARY KEY` form) ‚Äî the i16/i32/i64 type satisfies the key gate.

### 13.2 Conflicts at `CREATE TABLE` (all `42601`)

These mirror PG's `transformColumnDefinition`:

- **An explicit `DEFAULT` together with `IDENTITY`** is `42601 syntax_error` ("both default and
  identity specified for column *c* of table *t*"). Because a `serial` column *is* an explicit default
  (the synthesized `nextval`), **`serial ‚Ä¶ GENERATED ‚Ä¶ AS IDENTITY`** raises the *same* "both default
  and identity" error (PG-exact ‚Äî a serial column is desugared to a default before the identity check).
- **Two identity specifications** on one column (`‚Ä¶ GENERATED ALWAYS AS IDENTITY GENERATED BY DEFAULT
  AS IDENTITY`) is `42601` ("multiple identity specifications for column *c* of table *t*"), detected
  at parse (the second `GENERATED` clause).
- **Multiple identity columns in one table are allowed** (PG allows it ‚Äî like multiple `serial`
  columns); each gets its own owned sequence.

### 13.3 The `OVERRIDING` clause and INSERT value selection

`INSERT ‚Ä¶ [(cols)] [OVERRIDING { SYSTEM | USER } VALUE] (VALUES ‚Ä¶ | SELECT ‚Ä¶)`. The clause sits after
the optional column list and before the source. For each identity column the *effective* inserted
value is chosen per the table below (a value is "explicit" when supplied as a literal/param/SELECT
projection ‚Äî **not** the `DEFAULT` keyword and **not** an omitted column, both of which always use the
sequence):

| column kind | explicit value, no OVERRIDING | `OVERRIDING SYSTEM VALUE` | `OVERRIDING USER VALUE` |
|---|---|---|---|
| `GENERATED BY DEFAULT` | use the explicit value (sequence **not** advanced) | use the explicit value | **discard** it, use the sequence |
| `GENERATED ALWAYS` | **`428C9`** | use the explicit value | **discard** it, use the sequence |

- **`428C9 generated_always`** ("cannot insert a non-DEFAULT value into column *c*") fires when an
  explicit value targets a `GENERATED ALWAYS` column without `OVERRIDING SYSTEM VALUE`. It is a
  **statement-level analysis check**, raised *before* any row is materialized ‚Äî so it fires even on a
  `SELECT` source that returns **zero rows** (PG raises it during query rewrite). For a `VALUES` source
  it fires if **any** row supplies a non-`DEFAULT` value in that column (an all-`DEFAULT` column is
  fine).
- **`OVERRIDING USER VALUE`** discards the supplied value for *every* identity column and uses the
  sequence (advancing it), for both `ALWAYS` and `BY DEFAULT` ‚Äî implemented by treating the column as
  omitted (its `nextval` default applies).
- The clause is **accepted as a no-op** when the statement has no identity target (PG accepts it),
  including on a table with no identity columns at all.
- An explicit value supplied to a `GENERATED BY DEFAULT` column (no override) is stored **without**
  advancing the sequence ‚Äî exactly PG (the sequence and a later auto-value can then collide, as in PG).
- A NULL explicit value into a `BY DEFAULT` column (no override) is the ordinary `NOT NULL` path ‚Äî
  `23502` (the column is `NOT NULL`).

### 13.4 UPDATE of an identity column

- Assigning a **`GENERATED ALWAYS`** identity column an ordinary expression is `428C9` ("column *c*
  can only be updated to DEFAULT"). `UPDATE ‚Ä¶ SET c = DEFAULT` is the sole accepted form: it applies
  the synthesized `nextval(...)` default per matched row and advances the owned sequence, matching
  PostgreSQL.
- Assigning a **`GENERATED BY DEFAULT`** identity column is allowed (an ordinary value assignment) ‚Äî
  unless the column is also the `PRIMARY KEY`, where jed's standing PK-reassignment narrowing applies
  first (`0A000`, ¬ß11 step 6 of CLAUDE.md ‚Äî relaxable later; a documented divergence from PG, which
  would re-key). The ALWAYS `428C9` check is ordered **before** the PK narrowing, so an ALWAYS identity
  PK reports `428C9` (PG's code), not `0A000`.

### 13.5 Persistence & determinism

The two identity bits (`is_identity`, `identity_always`) live in the column entry's existing `flags`
byte (format_version 15, ¬ß3) ‚Äî no new field, no new catalog object. Everything else (the owned
sequence, the owner link, the `nextval` default) is exactly a `serial` column's bytes, so the on-disk
shape is `serial` + 2 flag bits. After a reopen the bits restore the INSERT/UPDATE gating. Identity
adds **no** new nondeterminism (it inherits ¬ß12.4 wholesale: a transactional owned sequence, a pure
auto-name, a deterministic auto-drop). Pinned by the `identity_table.jed` golden (¬ß3,
`rust == go == ts == ruby`) and the `ddl/identity.test` corpus.

## 14. `AS { smallint | integer | bigint }` ‚Äî the sequence data type (S5)

PostgreSQL sequences carry a **value data type** (`AS smallint | integer | bigint`, PG 10+). jed
adopts it. The type's *only* observable effect is on the **bounds**: it sets the default
`MINVALUE`/`MAXVALUE` and constrains explicit ones. Since jed's counter is already an `i64` and every
operation (`nextval`/`setval`/`RESTART`/`currval`) reads only the stored `(min_value, max_value)`, the
type is **fully reducible to those two fields** ‚Äî so jed derives the bounds at `CREATE` and **does not
persist the type** (see *Persistence* below). This closes the bigint-flavored divergence (decisions
3/9/11) with **no `format_version` change**.

### 14.1 The three types and their ranges

| `AS` type | aliases | range `[T_min, T_max]` |
|---|---|---|
| `smallint` | `int2`, `i16` | `[-32768, 32767]` |
| `integer` | `int`, `int4`, `i32` | `[-2147483648, 2147483647]` |
| `bigint` (default) | `int8`, `i64` | `[-9223372036854775808, 9223372036854775807]` (full `i64`) |

The `AS` argument is a type name resolved like any column type; a name that does not resolve to
`i16`/`i32`/`i64` (`AS text`, `AS numeric`, ‚Ä¶) is `22023 invalid_parameter_value` ("sequence type must
be smallint, integer, or bigint"). With no `AS` clause the type is **`bigint`** ‚Äî the prior default.
(NB: the `bigint` descending default `MINVALUE` is now `-2^63` = `i64::MIN`, matching PG 18 exactly;
the pre-S5 code used `-(2^63-1)`, a latent off-by-one against PG ‚Äî corrected here, and untested before,
so no golden moved for it.)

### 14.2 Bound derivation and validation (PG-exact, `init_params`)

Given the resolved type `T` with range `[T_min, T_max]`:

- **Defaults** (relative to `T`, not hard-coded `i64`): ascending `INCREMENT` ‚Üí `MINVALUE 1`, `MAXVALUE
  T_max`; descending ‚Üí `MINVALUE T_min`, `MAXVALUE -1`. `NO MINVALUE` / `NO MAXVALUE` select these
  same type defaults.
- **Validation order** (matching PG): `INCREMENT 0` and `CACHE < 1` first (`22023`); then an explicit
  `MAXVALUE > T_max` is `22023` ("MAXVALUE (x) is out of range for sequence data type *T*"); then an
  explicit `MINVALUE < T_min` is `22023` ("MINVALUE (x) is out of range ‚Ä¶"); then the consistency
  check `MINVALUE > MAXVALUE` (`22023`); then `START` ‚àà `[MINVALUE, MAXVALUE]` (`22023`). The
  type-range checks fire **before** the consistency check (oracle-pinned: `AS smallint MINVALUE 50000
  MAXVALUE 40000` reports the `MAXVALUE` range error, not `MINVALUE > MAXVALUE`).

### 14.3 `AS` is order-free; `serial` and identity auto-wire it

- On `CREATE SEQUENCE`, `AS T` is an **order-free option** (it may appear anywhere among
  `INCREMENT`/`MINVALUE`/‚Ä¶ ‚Äî PG allows `INCREMENT -1 AS smallint`), at most once.
- A **`serial`/`bigserial`/`smallserial`** column's owned sequence is built with `AS integer` /
  `AS bigint` / `AS smallint` respectively (¬ß12.2 step 4).
- A **`GENERATED ‚Ä¶ AS IDENTITY`** column's owned sequence is built with the **column type** as `AS T`
  (`i16` ‚Üí `smallint`, etc. ‚Äî ¬ß13.1). An explicit `AS` inside the identity `( ‚Ä¶ )` options is `42601`
  ("conflicting or redundant options") ‚Äî the column type already fixes the type. The identity `( ‚Ä¶ )`
  options are otherwise validated against the **column type's** range.

### 14.4 Persistence ‚Äî the type is *not* stored (a deliberate, recorded decision)

The `SequenceDef` on disk is **unchanged** (the six `i64` fields + flags ‚Äî ¬ß3); the `AS` type is baked
into `min_value`/`max_value` at `CREATE` and never stored separately. Rationale: (a) every runtime
operation depends only on the bounds, so the type carries no additional behavior to persist; (b) a
typed sequence and a plain `CREATE SEQUENCE ‚Ä¶ MINVALUE ‚Ä¶ MAXVALUE ‚Ä¶` with the same bounds are then
**byte-identical**, keeping the cross-core contract and the determinism story trivial; (c) it avoids a
`format_version` bump for a purely value-level change. The one thing this forecloses is `ALTER SEQUENCE
‚Ä¶ AS type` (which would re-derive the default bound for the *new* type and so needs the old type on
record) and type introspection ‚Äî both already deferred (`0A000` / no `pg_catalog`). If either lands,
it adds a persisted type field **then**, with a `format_version` bump at that point. The
`serial_table.jed` / `identity_table.jed` goldens move only because their integer column's owned
sequence now records `MAXVALUE 2147483647` (was `2^63-1`); `sequence_table.jed` (explicit-bounds /
ascending-bigint sequences) is byte-unchanged.

A second, smaller consequence surfaces in **S6's `ALTER SEQUENCE`** (¬ß15): an explicit
`MINVALUE`/`MAXVALUE` on `ALTER` cannot be range-checked against the sequence's *original* declared
type (it is gone), so jed bounds it only by the `i64` range, and `NO MINVALUE` / `NO MAXVALUE` reset the
*open* direction to the **bigint** bound rather than the original type's. PG (which keeps the type)
would reject an out-of-type-range `ALTER MAXVALUE` and reset to the type bound. This is a deliberate
extension of the type-not-persisted decision, divergent **only** for a typed (`smallint`/`integer`)
sequence altered along the open direction ‚Äî the common `bigint`/default sequence matches PG exactly.
The `ddl/alter_sequence.test` corpus therefore exercises only default-typed sequences; the typed-ALTER
divergence is recorded here, not oracle-tested.

## 15. `ALTER SEQUENCE` ‚Äî the definition-changing options + `RENAME TO` (S6)

S2 (¬ß4) landed `ALTER SEQUENCE ‚Ä¶ RESTART` as the lone action. S6 completes the `ALTER` surface that
needs **no on-disk change**: the **option set** (re-edit the definition) and **`RENAME TO`** (move the
catalog key). The two `ALTER` forms still off the table ‚Äî `AS type`, `OWNED BY`, `OWNER TO`,
`SET { SCHEMA | LOGGED | UNLOGGED }` ‚Äî stay `0A000` (`AS type` because the type is not persisted, ¬ß14.4;
the rest because jed has no schema/owner/storage-class concept). There is **no `format_version`
change**: the six `i64` fields + flags of a `SequenceDef` (¬ß3) and a column's `default_expr` text are
already variable catalog data, so no golden moves.

### 15.1 Surface and grammar

```sql
ALTER SEQUENCE [IF EXISTS] s
    [INCREMENT [BY] n] [MINVALUE m | NO MINVALUE] [MAXVALUE x | NO MAXVALUE]
    [START [WITH] s0] [RESTART [[WITH] r]] [CACHE c] [[NO] CYCLE]    -- ‚â•1 option, order-free
ALTER SEQUENCE [IF EXISTS] s RENAME TO s2
```

After the (optional `IF EXISTS`) name, the parser dispatches: `RENAME` ‚Üí the rename form; `OWNED` /
`OWNER` / `SET` ‚Üí `0A000`; otherwise the **option loop** ‚Äî the *same* order-free loop `CREATE SEQUENCE`
uses (`spec/grammar/grammar.ebnf`, `sequence_option`), extended to also accept `RESTART [[WITH] n]` as
an interleavable pseudo-option (valid only here, never in `CREATE` or an identity `( ‚Ä¶ )`). A repeated
option is `42601` ("‚Ä¶ specified more than once" / PG's "conflicting or redundant options"). The option
form requires **at least one** option (or `RESTART`); an empty `ALTER SEQUENCE s` is `42601` (PG: a
bare `ALTER SEQUENCE` is a syntax error). An `AS` token is *parsed* into the option set (it shares the
loop) and rejected at execution as `0A000` (`ALTER ‚Ä¶ AS type`).

### 15.2 Option execution ‚Äî PG `init_params(isInit = false)`

The existing definition is cloned; only **written** options overwrite their field; everything else ‚Äî
crucially `last_value` and `is_called` ‚Äî is preserved. Validation re-uses `CREATE`'s rules with three
`ALTER`-specific points (all oracle-pinned against `postgres:18`):

1. **`INCREMENT` sign does not recompute bounds.** Unlike `CREATE` (where a negative `INCREMENT`
   defaults `MAXVALUE -1`), an `ALTER` that only changes `INCREMENT` keeps the existing `MINVALUE`/
   `MAXVALUE`. `NO MINVALUE` / `NO MAXVALUE` *do* recompute the default for the (new) sign ‚Äî to the
   **bigint** bound on the open direction (¬ß14.4 divergence).
2. **`MINVALUE < MAXVALUE` is strict** ("MINVALUE (m) must be less than MAXVALUE (x)"). PG rejects
   `MINVALUE == MAXVALUE`; jed's shared validator uses `>=` (a latent `CREATE` divergence ‚Äî jed
   previously allowed `==` ‚Äî corrected in S6 for both paths; no golden/corpus moved since no fixture
   built a one-value sequence).
3. **Two post-edit cross-checks, in order:** first `START ‚àà [MINVALUE, MAXVALUE]` ("START value (s)
   cannot be less than MINVALUE (m)" / "greater than MAXVALUE (m)"); then the **preserved/‚Äãrestarted
   `last_value` ‚àà [MINVALUE, MAXVALUE]** ("RESTART value (lv) cannot be ‚Ä¶" ‚Äî PG's wording even with no
   `RESTART`). So narrowing `MAXVALUE` below the current value is `22023`, reported against the
   `last_value`. `RESTART` (if written) sets `last_value`/`is_called` *before* this second check.

`ALTER ‚Ä¶ <options>` is a catalog write (transactional, ¬ß5) and touches **no** session state.

### 15.3 `RENAME TO` and owned sequences

`RENAME TO s2` checks the relation namespace: if `s2` (lowercased) already names a table / index /
sequence ‚Äî **including `s` itself** ‚Äî it is `42P07` ("relation already exists"). Otherwise the entry
moves to the new key with `name = s2`; the counter, options, and `owned_by` link are unchanged.

The wrinkle jed's name-based `nextval` (¬ß4) introduces over PG's OID reference: an **owned** sequence
(¬ß12) is named by its owning column's stored `DEFAULT nextval('s')` expression *text*. So on a rename of
an owned sequence, jed **rewrites that column's default** to `nextval('s2')` (regenerating the
`expr_text` + re-parsing, exactly as `serial` synthesized it) in the same statement. After the rename a
subsequent `INSERT` advances the renamed sequence ‚Äî matching PG's observable behavior. The `owned_by`
pair (`table`, column-ordinal) is untouched, so `DROP TABLE` auto-drop (¬ß12.3) and the `2BP01` on a
direct `DROP SEQUENCE` of the renamed owned sequence both still hold. (A *plain* `DEFAULT nextval('s')`
on a non-owned sequence is **not** rewritten ‚Äî it has no `owned_by` link and PG makes no dependency
either, ¬ß10 decision 4 ‚Äî so renaming such a sequence then inserting raises `42P01`, exactly as in PG
where the un-tracked textual reference likewise breaks.)

### 15.4 Determinism

S6 adds no nondeterminism: the option edit and the rename are pure catalog mutations on the
transactional snapshot (¬ß5), the owned-column default rewrite is a deterministic function of
`(owner, new name)`, and nothing reads the clock or entropy. Cross-core byte-identity is unaffected
(no format change); the feature is pinned by the `ddl/alter_sequence.test` corpus (oracle-clean,
default-typed sequences) and the per-core unit tests for the deliberate divergences (the transactional
rollback of an `ALTER`, the read-only `25006`, and ‚Äî where reachable ‚Äî the typed-ALTER bound
divergence of ¬ß14.4).
